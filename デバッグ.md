# ALB から EC2 への HTTPS ヘルスチェック失敗のデバッグ記録

## 事象
AWS 側で `ALB -> EC2` の通信を `HTTPS:443` に変更した後、ALB のヘルスチェックが失敗した。

ヘルスチェック設定は以下。
- プロトコル: `HTTPS`
- ポート: `443`
- パス: `/api/health`

## 当初の API 資材の状態
API 資材の構成は以下だった。
- `api` コンテナ: Fastify が内部 `3000` 番で待ち受け
- `nginx` コンテナ: 外部 `80` 番を公開し、`api:3000` にリバースプロキシ

該当箇所:
- `docker-compose.yml`: `80:80` を公開
- `nginx/default.conf`: `listen 80;`
- `src/routes/health.ts`: `/api/health` は実装済み

この状態では、ALB から EC2 へ `HTTPS:443` で接続しても、EC2 側に `443` の受け口がないためヘルスチェックは成功しない。

## 実施した修正
API 資材を以下のように変更した。

### 1. nginx の公開ポートを 443 に変更
`docker-compose.yml` を修正。
- 変更前: `80:80`
- 変更後: `443:443`

### 2. nginx を HTTPS 待ち受けに変更
`nginx/default.conf` を修正。
- 変更前: `listen 80;`
- 変更後: `listen 443 ssl;`

加えて以下を設定。
- `ssl_certificate /etc/nginx/certs/server.crt;`
- `ssl_certificate_key /etc/nginx/certs/server.key;`
- `ssl_protocols TLSv1.2 TLSv1.3;`

### 3. 証明書配置場所を追加
以下の配置前提にした。
- `nginx/certs/server.crt`
- `nginx/certs/server.key`

補足:
- ALB からターゲットへの HTTPS 通信では、自己署名証明書でも動作可能
- ACM 証明書は ALB 用であり、EC2 上の nginx にそのまま配置する用途では使えない

## ハマったポイント
修正後すぐはヘルスチェックがまだ失敗していた。原因は 2 つあった。

### 1. 証明書ファイルが存在しなかった
`nginx/certs/` に実ファイルがなく、`server.crt` / `server.key` が未配置だった。
そのため、`nginx` が `443 ssl` で正常起動できない状態になり得た。

EC2 上で自己署名証明書を生成して配置した。

```bash
cd /path/to/dense-deck-api
mkdir -p nginx/certs
openssl req -x509 -nodes -newkey rsa:2048 \
  -keyout nginx/certs/server.key \
  -out nginx/certs/server.crt \
  -days 365 \
  -subj "/CN=localhost"
```

### 2. nginx コンテナが古いままだった
`docker ps` を確認すると、`dense-deck-nginx` が 2 か月前に起動した古いコンテナのままで、まだ `0.0.0.0:80->80/tcp` で動いていた。

つまり、ローカルファイルは直っていても、EC2 上の実コンテナは新設定に切り替わっていなかった。

対応として、コンテナを作り直した。

```bash
cd /path/to/dense-deck-api
docker-compose down
docker rm -f dense-deck-nginx 2>/dev/null || true
docker compose up -d --build
```

## 確認結果
再作成後の `docker ps` では以下になった。
- `dense-deck-nginx`: `0.0.0.0:443->443/tcp`
- `dense-deck-api`: `3000/tcp`

EC2 上で以下を確認した。

```bash
curl -vk https://localhost/api/health
```

結果:
- TLS ハンドシェイク成功
- `HTTP/1.1 200 OK`
- レスポンス: `{"ok":true,"service":"dense-deck-api"}`

ここまでで、EC2 側の `HTTPS:443 /api/health` は正常動作と確認できた。

## 最終結論
今回のヘルスチェック失敗の原因は、主に以下。
- API 資材が当初 `80` 待ち受けのままで、`443` の HTTPS 終端に未対応だった
- `nginx` 用の証明書・秘密鍵が未配置だった
- EC2 上の `nginx` コンテナが古い設定のままで再作成されていなかった

最終的な対応は以下。
- `docker-compose.yml` を `443` 公開に修正
- `nginx/default.conf` を `listen 443 ssl;` に修正
- `server.crt` / `server.key` を配置
- `docker compose up -d --build` で `nginx` を新設定に入れ替え
- `curl -vk https://localhost/api/health` で `200 OK` を確認

## 補足
EC2 ローカルで `HTTPS:443 /api/health` が成功しているため、以後 ALB 側でヘルスチェックが失敗する場合は、AWS 側設定を確認する。
主な確認項目は以下。
- Target Group の protocol が `HTTPS`
- Health check protocol が `HTTPS`
- Health check port が `443`
- Health check path が `/api/health`
- Success codes が `200`
- EC2 の Security Group で、ALB の Security Group から `443/tcp` を許可している

## 追加で判明した最終原因
当初は `ALB -> EC2` 側の HTTPS 化に伴う問題を中心に切り分けていたが、最終的には `CloudFront -> ALB` の入口でも問題があった。

症状としては以下だった。
- ALB のヘルスチェックは成功している
- EC2 上で `curl -vk https://localhost/api/health` は成功する
- しかしブラウザから CloudFront 経由の `/api/auth/me` は `504 Gateway Timeout`
- ALB 側のモニタリングに `RequestCount` などの値がほぼ入っていない

このことから、CloudFront から ALB に届く前段で落ちている可能性が高いと判断した。

## CloudFront から ALB への直接疎通確認
外部端末から以下を確認した。

```bash
curl -vk https://origin.densedeckpoker.com/api/health
```

結果:
- DNS 解決はできる
- ALB の IPv4 アドレスには到達しようとする
- しかし `443` 接続がタイムアウトする

この結果から、`origin.densedeckpoker.com -> ALB:443` の入口で遮断されていると判断した。

## 最終原因
ALB の Security Group インバウンドルールで、`HTTPS:443` の送信元として設定していた CloudFront の Prefix List が誤っていた。

当初設定していたものは、CloudFront の origin-facing 用として不適切な Prefix List であり、実際の CloudFront からの接続を ALB が受け付けられていなかった。

そのため以下の状態になっていた。
- CloudFront -> ALB で接続失敗
- ALB にリクエストが到達しない
- CloudFront が `504 Gateway Timeout` を返す
- ALB 側モニタリングにも値がほぼ入らない

## 対応
ALB の Security Group のインバウンドルールを修正し、`HTTPS:443` の送信元を正しい CloudFront origin-facing の Prefix List に変更した。

最終的に使用した Prefix List:
- `com.amazonaws.global.cloudfront.origin-facing`

画面上では以下のような値で確認できた。
- `pl-58a04531 (com.amazonaws.global.cloudfront.origin-facing)`

## 対応後の結果
Prefix List を正しいものに変更した後、CloudFront 経由のアクセスが正常化した。

つまり今回の最終的な問題は、API 資材だけではなく AWS 側の Security Group 設定にもあった。

## 最終まとめ
今回の障害は、以下が複合して発生していた。

- API 資材が当初 `80` 待ち受けで、`ALB -> EC2 = HTTPS:443` に未対応だった
- nginx 用の証明書・秘密鍵が未配置だった
- EC2 上の nginx コンテナが古い設定のままで再作成されていなかった
- ALB のターゲットグループやルーティング設定の見直しが必要だった
- 最後に、ALB の Security Group で CloudFront origin-facing 用 Prefix List の指定が誤っていた

最終的には、API 資材修正に加えて AWS 側の以下を揃えることで解消した。
- `ALB -> EC2 = HTTPS:443`
- EC2 SG で `ALB SG -> 443` を許可
- ALB SG で `CloudFront origin-facing Prefix List -> 443` を許可
- その Prefix List は正しい `com.amazonaws.global.cloudfront.origin-facing` を使う
